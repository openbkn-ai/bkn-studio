# BKN Studio 智能体整合章程

## 目的

本章程用于约束 `bkn-studio` 中的模块如何作为“可被其他智能体整合引用的能力”被设计、暴露和演进。

目标不是让其他智能体直接复用路由页，而是让它们通过稳定、可受控、可审计的模块能力接入。

## 适用范围

以下场景必须遵守本章程：

- 模块将来会被别的智能体组合进更大流程
- 模块场景需要被宿主或其他业务容器复用
- 模块需要暴露稳定输入输出，而不是只在当前路由里运行

## 核心原则

### 1. 页面不是复用单元

- `pages/*` 只作为路由页
- 不允许把页面文件当作跨智能体复用入口
- 路由页应该尽量薄，只负责把路由接到场景上

### 2. 场景才是复用单元

- `scenes/*` 承担完整业务场景
- 其他智能体默认应引用 `scene`
- `scene` 应表达一段完整业务能力，而不是零散控件

### 3. 模块必须有稳定出口

- 每个模块必须通过 `index.ts` 统一公开
- 不允许外部智能体深层 import 模块内部私有文件
- 公开内容应尽量稳定、可预期

### 4. 契约先于实现

- `contracts/*` 先定义场景输入输出
- `scene`、`service`、`types` 再围绕契约实现
- 不允许“先写死实现，再让外部去猜怎么接”

### 5. 清单必须可读

- `module.manifest.ts` 是模块能力清单
- 其他智能体应优先通过 manifest 判断该模块是否可整合

## 标准结构

```text
src/modules/<module-name>/
  components/
  contracts/
  pages/
  scenes/
  services/
  types/
  index.ts
  module.manifest.ts
  routes.tsx
```

## 各层职责

### `pages`

负责：

- 路由参数读取
- 权限挂载
- 壳层接入

不负责：

- 成为对外复用入口
- 承担复杂场景编排

### `scenes`

负责：

- 可复用业务场景
- 场景级状态
- 场景级事件编排

要求：

- 输入通过 `props`
- 输出通过回调
- 允许外部控制关键行为
- 尽量不要把导航、消息、确认框写死

### `components`

负责：

- 场景内细粒度 UI 块
- 复杂表格、筛选区、详情分组、表单区块

要求：

- 默认视为模块内部组件
- 只有明确需要时才从 `index.ts` 暴露为公开组件

### `contracts`

负责：

- 定义场景输入
- 定义场景输出
- 定义外部可依赖的数据模型

要求：

- 语义稳定
- 不把内部实现细节泄露为对外契约

### `module.manifest.ts`

负责：

- 描述模块能力边界
- 供人和智能体快速判断模块可用性

至少包含：

- 模块 id
- 模块名称
- 可暴露场景
- 场景输入摘要
- 权限依赖
- 后端依赖
- 是否要求壳层
- 是否支持嵌入模式
- 是否支持只读模式

### `index.ts`

负责：

- 统一公开模块可复用内容

可导出：

- `scenes`
- `contracts`
- `types`
- `manifest`
- 明确允许外部使用的组件

禁止：

- 把整个模块所有内部文件无差别导出

## 场景设计规则

### 输入规则

- 明确最小必要输入
- 避免把当前路由对象、全局上下文对象整个传进去
- 输入优先传业务语义，而不是框架细节

### 输出规则

- 用回调暴露关键事件
- 例如：
  - `onSelect`
  - `onOpenDetail`
  - `onCreate`
  - `onSubmitSuccess`

### 行为替换规则

场景中以下行为应尽量允许宿主替换：

- 跳转
- 消息提示
- 确认框
- 数据刷新触发

### 壳层依赖规则

场景必须声明：

- 是否必须运行在 `console shell`
- 是否支持无壳层嵌入
- 是否要求顶部工具栏插槽

## 对外依赖规则

### 允许

- 依赖 `framework/*`
- 依赖本模块内部 `components/*`
- 依赖本模块 `services/*`

### 禁止

- 深层依赖其他模块内部文件
- 直接依赖其他模块私有组件
- 把路由页作为场景依赖

## 变更管理规则

以下改动视为“对外能力改动”：

- 修改 `index.ts`
- 修改 `contracts/*`
- 修改 `module.manifest.ts`
- 修改已公开 `scene` 的输入输出

这类改动必须：

- 在 PR 中明确说明影响范围
- 说明是否会影响其他智能体整合
- 经人工 review 后再合并

## 验收清单

一个模块要被视为“可被其他智能体整合引用”，至少需要满足：

- 有 `scene`
- 有 `contracts`
- 有 `manifest`
- 有统一 `index.ts`
- 没有要求外部深层 import 私有文件
- 场景输入输出清晰
- 场景不强依赖当前路由页才能运行
