# 系统管理页面模板落地计划

## 1. 目标

把当前系统管理模块已经验证有效的页面结构、视觉规则和交互规则，沉淀为后续可复用的页面模板，减少页面风格漂移和重复实现。

## 2. 范围

### 本次纳入

- 系统管理标准列表页模板
- 系统管理标准新建页模板
- 系统管理标准抽屉页模板
- 分页与页面状态管理规范
- 当前授权管理页作为首个模板化整改样板

### 本次不纳入

- 全站所有模块的全面样式替换
- 前端设计系统全面重构
- 所有旧页面一次性迁移

## 3. 实施阶段

### Phase 1：模板抽取

目标：

- 从系统管理现有实现中抽取共性页面结构

产出：

- 模板规范文档
- 模板参考实现清单

### Phase 2：授权管理样板页收口

目标：

- 把授权管理列表页、新建页、抽屉页整理成标准模板样板

产出：

- 稳定的授权管理三页
- 公共分页状态问题修复

### Phase 3：系统管理模块内对齐

目标：

- 让用户管理、角色管理、授权管理、日志管理四个模块在结构和风格上统一

产出：

- 统一的操作栏
- 统一的表格
- 统一的分页
- 统一的抽屉/详情层次

### Phase 4：向其他后台模块扩展

目标：

- 以系统管理模板为基准，逐步用于数据连接、数据目录等后台模块

产出：

- 页面模板复用策略
- 模块级迁移顺序建议

## 4. 技术策略

- 样式基于 `BusinessPageTemplate.module.css` 扩展
- 模块局部样式统一收敛到模块级样式文件
- 分页统一走 `TablePaginationBar`
- 页面分页/关键词状态统一走 `use-page-state`
- 避免在单一页面中叠加多套互斥视图模式

## 5. 风险点

1. 页面已存在历史实现差异
   - 风险：迁移时看似统一，细节仍漂移
   - 应对：以模板参考实现逐个对齐，不做口头约定

2. 公共钩子被错误使用
   - 风险：再次出现持续重渲染
   - 应对：公共钩子返回稳定回调，并在模板文档中明确限制

3. 团队成员继续各自实现
   - 风险：后续新页面再次偏离模板
   - 应对：把模板写入 `specs`，并作为后续开发默认要求

## 6. 建议执行顺序

1. 先固定系统管理模板
2. 再用模板修正系统管理剩余页面细节
3. 然后迁移到数据资源类后台页面
