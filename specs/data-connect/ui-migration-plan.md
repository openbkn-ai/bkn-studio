# 数据连接页面效果迁移计划

## 目标

将 `bkn-studio` 中的 `data-connect` 工作区页面效果调整为接近旧 `vega` 数据连接页的布局与交互节奏，但不复用旧前端的接口层和项目私有封装。

本次迁移的固定前提是：

- 前端工程结构继续遵守 `bkn-studio` 当前分层
- 后端接口继续对齐最新 `vega-backend`
- 迁移的是页面效果、布局和交互组织方式
- 不迁移旧 `vega` 的接口模型、权限封装和历史包袱

## 参考来源

- 线上参考页：
  - `dip-hub/business-network/vega/data-connect`
- 旧前端源码：
  - `D:\kowell\kowell-dip\web\apps\vega\src\pages\DataConnect`
  - `D:\kowell\kowell-dip\web\apps\vega\src\services\dataConnect`
  - `D:\kowell\kowell-dip\web\apps\vega\src\services\scanManagement`

## 迁移边界

### 迁移内容

- 数据连接列表页的工作区布局
- 页面内部操作栏和筛选栏结构
- 表格区域的留白、密度、层次
- 详情抽屉的信息分组
- 新建 / 编辑页面的步骤体验
- 扫描管理页的计划区与任务区布局
- 页面内空态、错误态、加载态的落位方式

### 不直接迁移的内容

- 旧 `/api/data-connection/v1/*` 接口
- 旧 `services/*` 的实现
- 旧项目私有组件体系
- 旧权限判断逻辑
- Excel 元数据能力整套逻辑
- 与当前 `bkn-studio` 架构冲突的页面组织方式

## 后端接口基线

本次迁移中，页面效果可以参考旧 `vega`，但数据语义一律按最新 `vega-backend`：

- 数据连接实例：`catalogs`
- 连接器类型：`connector-types`
- 扫描计划：`discover-schedules`
- 扫描任务：`discover-tasks`
- 手动触发扫描：`POST /catalogs/:id/discover`

## 四步计划

### 第 1 步：确定迁移边界

目标：

- 明确旧页哪些部分只做视觉参考
- 明确哪些逻辑必须继续使用当前实现

输出：

- 一份迁移范围说明
- 一份接口映射清单
- 一份页面拆分清单

### 第 2 步：迁移数据连接主页面效果

目标：

- 先把列表页工作区做成接近旧 `vega` 的结构
- 保持当前列表、搜索、筛选、分页、详情、测试、启停、删除能力不变

重点：

- 页面内操作栏
- 页面内表格容器
- 行内操作密度
- 页面留白和视觉层次

### 第 3 步：迁移扫描管理与表单页效果

目标：

- 让扫描页、新建页、编辑页和详情抽屉风格一致
- 保留当前已经接通的新接口行为

重点：

- 扫描计划区与任务区布局
- 新建 / 编辑页步骤结构
- 抽屉信息分组和字段呈现

### 第 4 步：细节收口与验收

目标：

- 让迁移结果不只是“像”，还要稳定可用

重点：

- 状态标签、文案、按钮排列
- 错误态、空态、加载态
- 自测和构建验证

## 页面拆分建议

### 列表页

- 页面内部操作栏
- 搜索和筛选区
- 表格区
- 行内操作区
- 详情抽屉

### 表单页

- 类型选择步骤
- 动态配置步骤
- 底部动作区
- 风险提示区

### 扫描管理页

- 扫描计划区
- 扫描任务区
- 任务详情抽屉
- 手动触发入口

## 风险

### 风险 1：旧页视觉依赖旧组件体系

处理方式：

- 只迁移版式和交互节奏
- 不直接复制旧 DOM 结构和旧组件

### 风险 2：旧页字段语义与新后端不完全一致

处理方式：

- 页面内仍使用 `data-connect` 业务语义
- 在 service 层统一做 DTO 映射

### 风险 3：为追求 1:1 还原导致重新引入旧包袱

处理方式：

- 优先迁移高感知区域
- 不追求全部细节逐项照搬

## 验收标准

- 页面整体结构接近旧 `vega` 数据连接页
- 用户在工作区内的操作路径更接近旧页习惯
- 所有数据继续来自最新 `vega-backend`
- 当前已实现功能不回退
- `pnpm lint`、`pnpm test -- --run`、`pnpm build` 通过
