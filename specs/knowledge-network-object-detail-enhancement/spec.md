# Issue 39 需求分析与规格说明

## 基本信息

- 来源: `openbkn-ai/bkn-studio` Issue #39
- 标题: `enhancement(knowledge-network): 业务对象详情页增强 — 数据视图、预览检索、SQL Skill、关系探索`
- 当前聚焦页面: `src/modules/knowledge-network/scenes/ObjectTypeDetailScene.tsx`

## 本次需求修正

基于最新约束，需求边界调整如下：

1. 一个对象类只会映射一个数据视图
2. 当前阶段只聚焦两块能力：
   - 对象类与数据视图的关系展示
   - 面向数据视图的数据查询能力
3. `Skill` 沉淀不纳入当前阶段，作为后续独立需求处理
4. 关系探索能力也不纳入当前阶段，后续再拆分

这意味着当前 Issue 39 需要被收敛为一个“对象详情数据工作台”子项目，而不是一次完成对象详情的全能力升级。

## 背景

当前业务对象详情页只覆盖以下内容：

1. 对象基础信息
2. 数据属性列表
3. 逻辑属性列表

但对象类已经具备数据源绑定语义，用户进入对象详情后，仍然无法直接回答两个核心问题：

- 这个对象类到底映射到哪个数据视图
- 这个数据视图里当前有哪些实际数据

因此当前阶段的目标很明确：

- 让对象详情页建立“对象类 -> 数据视图 -> 数据查询”的闭环

## 当前现状

### 前端现状

对象详情页实现位于：

- `src/modules/knowledge-network/scenes/ObjectTypeDetailScene.tsx`

当前已有能力：

- 展示对象名称、描述、标签、索引状态、概念组数量
- 展示数据属性
- 展示逻辑属性
- 支持对象编辑、删除

当前缺口：

- 未展示对象绑定的数据视图
- 未展示数据视图的 catalog / resource 信息
- 未集成数据视图样本预览
- 未提供对象详情内的数据查询入口

### 已有可复用能力

#### 1. 对象详情已返回单数据视图绑定

相关类型位于：

- `src/modules/knowledge-network/types/object-type.ts`

关键类型：

- `ObjectTypeDetail`
- `ObjectTypeDataSource`

现状判断：

- `ObjectTypeDetail` 已包含 `dataSource?: ObjectTypeDataSource`
- 当前可以明确按“单对象类 -> 单数据视图”建模，不需要设计多视图列表能力

#### 2. 已有资源预览服务

服务位于：

- `src/modules/knowledge-network/services/object-type-resource.service.ts`

可复用方法：

- `getObjectTypeResourcePreview`
- `listObjectTypeResourceFields`

说明：

- 数据预览基础能力已经存在
- 当前缺少的是把这部分能力接入对象详情页

#### 3. 已有对象详情页属性区块

对象详情页已有“数据属性 / 逻辑属性”区块，可以继续保留。

说明：

- 当前阶段不需要重构属性建模能力
- 重点是新增“数据视图”和“数据查询”两个业务区块

## 当前阶段需求范围

本次只拆为两个小需求。

### 子需求 A: 对象类与数据视图关系展示

目标：

- 在对象详情页清晰展示当前对象类绑定到哪个数据视图

最小交付：

- 展示当前对象类绑定的数据视图名称
- 展示数据视图所属 catalog / data source 标识
- 提供跳转到数据目录对应资源详情的入口

明确约束：

- 不考虑一个对象类绑定多个数据视图
- 不考虑主/辅数据源标记

空状态要求：

- 若对象类未绑定数据视图，页面明确提示“当前对象类未绑定数据视图”

### 子需求 B: 对象类对应数据视图的数据查询能力

目标：

- 用户在对象详情页内直接查看和查询该数据视图的数据

最小交付：

- 展示数据视图样本数据
- 支持关键字搜索 / 过滤
- 支持分页浏览样本数据

推荐交付：

- 默认样本预览使用已有预览服务能力
- 搜索优先使用前端过滤，降低接口改造成本
- 预览数据量受采样限制控制

边界说明：

- 当前阶段不做 SQL 编辑器
- 当前阶段不做“保存为 Skill”
- 当前阶段不做关系路径探索

## 当前阶段非目标

以下能力明确不在本次实现范围内：

- 多数据视图绑定
- SQL 检索工作台
- 查询结果保存为 Skill
- 对象关系探索
- 探索结果保存为 Logic / Skill

这些能力保留为后续独立需求。

## 页面结构建议

对象详情页建议整理为以下顺序：

1. 对象摘要
2. 数据属性 / 逻辑属性
3. 关联数据视图
4. 数据查询 / 数据样本预览

这样符合用户实际认知顺序：

- 先看对象定义
- 再看对象绑定的数据视图
- 最后看对象的实际数据

## 组件拆分建议

建议不要继续把新增逻辑全部堆在 `ObjectTypeDetailScene.tsx` 中。

推荐拆分：

- `ObjectTypeDetailScene`
  - 负责详情加载、整体布局
- `ObjectTypePropertyPanel`
  - 负责数据属性 / 逻辑属性现有区块
- `ObjectTypeBoundViewPanel`
  - 负责对象类与数据视图关系展示
- `ObjectTypeDataQueryPanel`
  - 负责样本数据预览、搜索、分页

## 服务与接口分析

### 当前已满足

- 对象详情可拿到 `dataSource`
- 可对具体资源做预览

### 当前需要确认

1. `ObjectTypeDataSource` 是否足以支持跳转到数据目录资源详情
2. 资源预览接口是否可直接用于对象详情页
3. 样本数据搜索是纯前端过滤，还是需要后端接口支持

## 验收口径

### 子需求 A 验收

- [ ] 打开对象类详情，可看到其绑定的数据视图
- [ ] 可看到基础资源标识信息
- [ ] 可跳转到数据目录对应资源详情
- [ ] 未绑定数据视图时有明确空状态

### 子需求 B 验收

- [ ] 可在对象详情页预览数据视图样本数据
- [ ] 可按关键字搜索 / 过滤样本数据
- [ ] 数据表格支持分页
- [ ] 预览失败时有错误提示

## 结论

Issue 39 当前阶段应被重定义为两个更小的需求：

1. 对象类与数据视图关系展示
2. 对象类的数据查询能力

这样可以在不引入 Skill、关系探索、复杂 SQL 工作台的前提下，优先建立对象详情页的数据闭环。
