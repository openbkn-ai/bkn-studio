# 数据连接与数据目录重构需求说明

## 1. 背景

当前数据侧模块在产品概念和页面职责上存在明显混用：

- `Catalog` 同时被当作连接对象、目录对象、资源容器来使用
- 扫描任务、资源列表、资源详情、数据查询、索引能力被堆叠在同一条页面链路中
- 用户难以判断当前是在管理连接，还是在管理数据资源

结合现有前端实现和 `D:\openbkn\bkn-foundry\adp\vega` 后端模型梳理，可以确认后端已经天然分成两层：

1. `Catalog`
   - 负责连接配置、启停、测试连接、扫描发现、扫描任务
   - 更适合作为“数据连接”概念
2. `Resource`
   - 负责承载扫描出的表、文件、API、索引、逻辑视图等资源
   - 更适合作为“数据目录”中的核心对象

因此本次需求不是单纯的页面美化，而是一次信息架构重构。

## 2. 目标

将当前混合模块重构为两类明确的业务菜单：

1. 数据连接
2. 数据目录

并统一后续治理能力围绕 `Resource` 开展，而不是围绕 `Catalog` 开展。

## 3. 核心概念映射

| 后端对象 | 产品概念 | 说明 |
|---|---|---|
| `Catalog` | 数据连接 | 一个物理连接或逻辑容器 |
| `DiscoverTask` | 扫描任务 | 针对连接执行的扫描/发现任务 |
| `Resource` | 数据资源 / 目录项 | 扫描出来的表、文件、API、索引、逻辑视图等 |
| `BuildTask` | 索引任务 | 针对资源执行的向量化/索引构建任务 |
| `Resource(category=logicview)` | 逻辑视图 | 仍然属于资源，不是新的一级根对象 |

## 4. 产品结构调整

### 4.1 一级菜单

#### A. 数据连接

职责边界：

- 新建连接
- 编辑连接
- 删除连接
- 测试连接
- 启停连接
- 手动扫描
- 扫描调度
- 扫描历史
- 扫描任务

不承担以下职责：

- 数据查询
- 数据向量化
- 数据质量
- 数据语义理解
- 逻辑视图运营

#### B. 数据目录

职责边界：

- 浏览资源列表
- 按连接、类型、状态筛选资源
- 查看资源详情
- 管理逻辑视图
- 进行数据查询
- 查看索引状态
- 承接质量、语义等资源治理能力

### 4.2 二级结构建议

```text
数据连接
  - 连接列表
  - 连接详情
  - 扫描任务

数据目录
  - 资源列表
  - 逻辑视图（可作为独立二级，也可作为资源类型筛选）
  - 资源详情
    - 基本信息
    - 字段 / Schema
    - 数据查询
    - 数据向量化
    - 数据质量
    - 数据语义理解
```

## 5. 页面职责重构

### 5.1 需要保留在“数据连接”的内容

- 连接基本信息
- 连接配置
- 测试连接
- 编辑连接
- 启停连接
- 立即扫描
- 扫描任务
- 扫描历史
- 扫描策略/调度

### 5.2 需要迁移到“数据目录”的内容

- 资源树主视图
- 资源列表
- 资源详情
- 逻辑视图详情
- 数据查询
- 数据向量化 / 索引构建
- 数据质量
- 数据语义理解

## 6. 路由重组建议

建议逐步收敛到如下路由结构：

```text
/studio/data-connections
/studio/data-connections/:catalogId
/studio/data-connections/:catalogId/discover-tasks

/studio/data-directory
/studio/data-directory/resources/:resourceId
/studio/data-directory/logic-views
/studio/data-directory/logic-views/:resourceId
```

如果要降低改动成本，也可先采用更保守的结构：

```text
/studio/data-connections
/studio/data-connections/:catalogId

/studio/data-directory
/studio/data-directory/:resourceId
```

## 7. 后端接口归位

### 7.1 数据连接侧

继续使用现有 `Catalog` 相关接口：

- `/catalogs`
- `/catalogs/:id`
- 连接测试、启停、删除
- discover task 相关接口

### 7.2 数据目录侧

继续使用现有 `Resource` 相关接口：

- `/resources`
- `/resources/:id`
- 资源详情、更新、删除
- build task 相关接口
- 资源预览 / 查询接口

## 8. 后端适配建议

本次不建议新增新的后端根模型，例如 `DataSource`。

原因：

- 现有 `Resource` 已经承载物理资源和逻辑视图
- 索引任务已经绑定 `ResourceID`
- 继续新增根模型会造成概念重叠

建议只补强“目录视角”的查询能力：

1. 资源列表增强筛选
   - 按 `catalog_id`
   - 按 `category`
   - 按 `status`
   - 按索引状态
   - 按关键词
2. 资源详情聚合信息增强
   - 基本信息
   - schema
   - discover status
   - index/build 状态摘要
   - 来源连接摘要
3. 逻辑视图单独筛选
   - 本质仍然查询 `Resource(category=logicview)`

## 9. 实施原则

### 9.1 不新增概念债务

- 不再把 `Catalog` 对用户表述为“数据目录”
- 不再把连接和资源治理混在一个页面里

### 9.2 治理能力全部围绕 `Resource`

包括但不限于：

- 数据查询
- 数据向量化
- 数据质量
- 数据语义理解

### 9.3 逻辑视图不是新的一级系统

逻辑视图仍然是资源的一种，只是在产品层需要更清晰的入口和筛选方式。

## 10. 分阶段落地建议

### Phase 1：概念与导航归位

- 新增“数据连接”一级菜单
- 新增“数据目录”一级菜单
- 调整文案，弱化 `Catalog` 对终端用户的暴露

### Phase 2：页面职责拆分

- 连接页只保留连接与扫描
- 资源页承接资源列表、详情、查询
- 将当前 catalog 页面中的资源明细能力迁出

### Phase 3：资源治理能力归位

- 数据查询接入资源详情
- 数据向量化接入资源详情
- 数据质量接入资源详情
- 数据语义理解接入资源详情

### Phase 4：高级对象后置处理

- dataset
- 更复杂的逻辑视图关系
- skills 沉淀与复用

## 11. 验收标准

- 用户能够明确区分“数据连接”和“数据目录”
- 扫描任务仅在数据连接侧出现
- 数据查询、索引、质量、语义能力仅在资源侧出现
- 逻辑视图不再作为独立根模型与资源体系并列
- 前端页面结构与后端 `Catalog / Resource` 两层模型保持一致
