# 数据连接与数据目录重构实施方案

## 1. 实施目标

围绕后端既有的 `Catalog` 与 `Resource` 模型，完成前端菜单、路由、页面职责的归位，避免继续在现有模块上叠加概念负担。

## 2. 实施范围

### 本次纳入

- 菜单重命名与重组
- 路由结构调整
- catalog 页面职责收敛
- resource 页面职责提升
- 扫描任务归位到连接侧
- 数据查询、向量化、质量、语义能力归位到资源侧

### 本次不纳入

- 新建后端根模型
- 大规模改造数据库结构
- skills 能力沉淀
- dataset 高阶产品化能力

## 3. 分阶段方案

### Phase 1：信息架构调整

目标：

- 明确“数据连接”和“数据目录”两个一级菜单
- 统一文案口径

交付：

- 菜单树调整方案
- 面包屑与页面标题调整方案
- 路由迁移清单

### Phase 2：连接侧页面收敛

目标：

- 将 catalog 页面收敛为连接管理页面

交付：

- 连接列表
- 连接详情
- 扫描任务/扫描历史

移除项：

- 资源树主视图
- 数据查询
- 资源治理动作

### Phase 3：目录侧页面建立

目标：

- 建立以 `Resource` 为核心的数据目录页面

交付：

- 资源列表页
- 资源详情页
- 逻辑视图筛选或二级入口

### Phase 4：资源治理能力接入

目标：

- 将围绕资源的治理能力全部纳入资源详情体系

交付：

- 数据查询
- 数据向量化
- 数据质量
- 数据语义理解

## 4. 技术策略

### 前端策略

- 以页面职责拆分为主，不先做大规模抽象
- 优先复用现有资源详情、查询、索引相关组件和服务
- 避免继续在单个 scene 中叠加多类业务逻辑

### 后端策略

- 继续复用现有 `/catalogs` 与 `/resources` 接口族
- 仅在必要时补充资源列表筛选和资源详情聚合信息
- 不引入新的“datasource”根实体

## 5. 风险点

1. 现有页面存在较强耦合
   - 风险：拆分过程中回归范围较大
   - 应对：先做菜单和路由层拆分，再逐步迁内容

2. 前端对 `Catalog` 概念依赖过深
   - 风险：仍然在新页面里透出旧语义
   - 应对：统一页面命名、文案、空状态、操作入口

3. 资源详情接口聚合信息不足
   - 风险：资源页仍需拼装过多请求
   - 应对：按需补充资源详情摘要字段

## 6. 产出顺序建议

1. 先出菜单树和路由清单
2. 再做连接侧页面收敛
3. 再建目录侧资源列表和详情
4. 最后迁查询、索引、质量、语义能力
